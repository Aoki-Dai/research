## 考察

- 経歴とスキルの乖離: 経歴書に書かれる文章（プロジェクト内容など）と、スキルセットでは、捉えている特徴が大きく異なることが示唆される。
- 統合モデルの挙動: 単純なテキスト結合では、文章量の多い「経歴」情報が支配的になり、「スキル」情報が埋もれてしまっている可能性がある。スキルを重視したい場合は、重み付けの調整や、別の統合手法（ベクトルの加重平均など）の検討が必要である。

## 定性分析（マッチング理由の解明）

プロジェクトとマッチした人材のテキストデータから名詞を抽出し、共通するキーワード（積集合）を分析しました。

### 分析結果の例

**プロジェクト1: 旅行予約システム設計**
（キーワード: テスト, 作成, 保守, 構築, デプロイ, 予約, 運用, システム設計, ドキュメント, 旅行...）

| マッチング手法 | 上位人材 | 共通キーワード数 | 主な共通キーワード | 考察 |
| :--- | :--- | :--- | :--- | :--- |
| **経歴** | User 204 | **7語** | テスト, 構築, システム設計, 導入, インフラ, プロジェクト, 自動 | 「システム設計」「構築」「テスト」といった**業務内容（コンテキスト）**レベルでの強い一致が見られる。 |
| **スキル** | User 288 | **0語** | なし | キーワードの直接的な重複はない。スキルマッチングはWord2Vecのベクトル類似度に基づいているため、単語が完全に一致しなくても意味的に近い（例: Java ≒ バックエンド開発）と判断されている可能性があるが、説明可能性は低い。 |
| **両方** | User 204 | **7語** | テスト, 構築, システム設計, 導入, インフラ, プロジェクト, 自動 | 経歴マッチングと同様の結果。経歴情報のテキスト量が多いため、統合ベクトルにおいても経歴情報が支配的になっていることが裏付けられた。 |

### 定性的な知見

1. **経歴マッチングの納得感**: プロジェクトの記述と人材の経歴記述の間で、「システム設計」「構築」「テスト」といった具体的なアクションやフェーズに関する用語が共通しており、なぜマッチしたかが直感的に理解しやすい。
2. **スキルマッチングの性質**: キーワードの直接一致が少なく、ブラックボックス的な挙動を示している。プロジェクト記述に特定の技術名（Java, Vueなど）が明記されていない場合でも、文脈から推測してマッチングしている可能性があるが、検証が難しい。
3. **統合モデルの課題**: 定量分析（Jaccard係数）の結果と同様に、定性分析でも「両方」の結果は「経歴」の結果と酷似しており、スキル情報が十分に活用されていない可能性が高い。
